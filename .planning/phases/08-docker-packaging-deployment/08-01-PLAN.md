**Create Dockerfile** (multi-stage build for the app container):

Stage 1 -- `builder`:
- Base image: `python:3.14-slim` (matches project Python version; multi-arch AMD64+ARM64)
- `WORKDIR /app`
- Install build-time system deps needed for Python package compilation: `apt-get update && apt-get install -y --no-install-recommends gcc build-essential && rm -rf /var/lib/apt/lists/*`
- Copy `requirements-docker.txt` and run `pip install --no-cache-dir -r requirements-docker.txt`

Stage 2 -- runtime:
- Base image: `python:3.14-slim`
- `WORKDIR /app`
- Install runtime system dep: `apt-get update && apt-get install -y --no-install-recommends curl && rm -rf /var/lib/apt/lists/*` (curl needed for HEALTHCHECK)
- Copy installed Python packages from builder: `COPY --from=builder /usr/local/lib/python3.14/site-packages /usr/local/lib/python3.14/site-packages` and `COPY --from=builder /usr/local/bin /usr/local/bin`
- Copy application code: `COPY web.py cli.py ask.py ./` and `COPY pipeline/ ./pipeline/`
- Copy pre-built data: `COPY data/chroma/ ./data/chroma/` (the vector store) and `COPY data/eval/ ./data/eval/` (evaluation golden queries)
- Copy source PDFs and manifests: `COPY sources/originals/ ./sources/originals/` and `COPY sources/manifests/ ./sources/manifests/`
- Copy entrypoint: `COPY docker/entrypoint.sh /entrypoint.sh` and `RUN chmod +x /entrypoint.sh`
- `EXPOSE 8080`
- HEALTHCHECK directive: `HEALTHCHECK --interval=15s --timeout=5s --start-period=30s --retries=3 CMD curl -sf http://localhost:8080/api/health || exit 1`
- `ENTRYPOINT ["/entrypoint.sh"]`

**Create Dockerfile.ollama** (custom Ollama image with pre-baked models):

Stage 1 -- `model-builder`:
- Base image: `ollama/ollama:latest`
- Install curl for readiness polling: `RUN apt-get update && apt-get install -y --no-install-recommends curl && rm -rf /var/lib/apt/lists/*`
- Start Ollama server, wait for readiness via curl polling loop (NOT fixed sleep), then pull both models:
```dockerfile
RUN ollama serve & \
    for i in $(seq 1 30); do \
        curl -sf http://localhost:11434/api/tags > /dev/null 2>&1 && break; \
        sleep 1; \
    done && \
    ollama pull llama3.1:8b && \
    ollama pull nomic-embed-text
```
Use curl polling loop per RESEARCH.md Open Question 1 recommendation (more reliable than `sleep 5` across hardware).

Stage 2 -- final image:
- Base image: `ollama/ollama:latest`
- Install curl for health checks: `RUN apt-get update && apt-get install -y --no-install-recommends curl && rm -rf /var/lib/apt/lists/*`
- Copy pre-pulled models: `COPY --from=model-builder /root/.ollama /root/.ollama`
- `EXPOSE 11434`
- HEALTHCHECK: `HEALTHCHECK --interval=10s --timeout=5s --start-period=10s --retries=5 CMD curl -sf http://localhost:11434/api/tags || exit 1`
- `CMD ["serve"]`

**DEPL-03 compliance note:** DEPL-03 requires "Ollama container automatically pulls default models on first startup." This is satisfied by baking models into the image at Docker build time -- the models are immediately available with zero user intervention, and the system works fully offline. Build-time baking is strictly superior to runtime pulling for our offline-first use case.

**IMPORTANT per Pitfall 1 from RESEARCH.md:** The official ollama/ollama image does NOT include curl. It MUST be installed explicitly for both the build polling loop and the runtime health check.

**IMPORTANT per Pitfall 7 from RESEARCH.md:** Use consistent `/root/.ollama` path for model storage. The multi-stage COPY approach avoids path mismatch between build and runtime.

**Create docker/entrypoint.sh** (app container entrypoint with Ollama health gating and pipeline initialization):

First, create the `docker/` directory.

The entrypoint has two responsibilities: (1) wait for Ollama readiness, and (2) start the app with full pipeline initialization.

**Why an inline Python wrapper is needed:** web.py calls `build_source_map()`, `retrieve.init()`, and `gen.init()` only in its `if __name__ == "__main__"` block. Running `uvicorn web:app` directly skips that block entirely, resulting in a broken system with no initialized retrieval pipeline. The solution is an inline Python script that imports web.py, calls the initialization functions in-process, then starts uvicorn -- all within the same Python process so module-level state persists.

```bash
#!/bin/bash
set -e

OLLAMA_URL="${OLLAMA_HOST:-http://ollama:11434}"

echo "SurvivalRAG: Waiting for Ollama at $OLLAMA_URL..."
MAX_RETRIES=60
for i in $(seq 1 $MAX_RETRIES); do
    if curl -sf "$OLLAMA_URL/api/tags" > /dev/null 2>&1; then
        echo "SurvivalRAG: Ollama is ready."
        break
    fi
    if [ "$i" -eq "$MAX_RETRIES" ]; then
        echo "SurvivalRAG: WARNING - Ollama not available after ${MAX_RETRIES}s, starting anyway..."
    fi
    sleep 1
done

echo "SurvivalRAG: Starting application with pipeline initialization..."
exec python -c "
import os
os.environ.setdefault('SURVIVALRAG_PORT', '8080')
from web import build_source_map, app
import pipeline.retrieve as retrieve
import pipeline.generate as gen
import uvicorn
import logging
logging.basicConfig(level=logging.INFO, format='%(asctime)s [%(levelname)s] %(name)s: %(message)s')
build_source_map()
retrieve.init(chroma_path='./data/chroma')
gen.init()
uvicorn.run(app, host='0.0.0.0', port=int(os.environ.get('SURVIVALRAG_PORT', '8080')))
"
```

Design decisions for entrypoint:
- Uses `OLLAMA_HOST` env var (default `http://ollama:11434` -- the Docker Compose service name) per CONTEXT.md locked decision
- 60-second timeout with 1-second polling (generous for slow hardware)
- On timeout: starts anyway with WARNING (not hard exit) so the web UI can show degraded status via /api/health
- Uses `exec` to replace shell process with the Python wrapper (proper signal handling, PID 1)
- Port 8080 per CONTEXT.md locked decision (NOT web.py's default 7860)
- Inline Python wrapper calls `build_source_map()`, `retrieve.init()`, and `gen.init()` in the same process before starting uvicorn, matching web.py's `__main__` block behavior
