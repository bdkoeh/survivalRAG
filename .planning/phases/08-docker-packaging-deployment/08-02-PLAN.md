**Create docker-compose.yml** (Docker Compose v2 specification, CPU-only default):

```yaml
# SurvivalRAG - Docker Compose Configuration
# Usage: docker compose up
# GPU:   docker compose -f docker-compose.yml -f docker-compose.gpu.yml up

services:
  ollama:
    build:
      context: .
      dockerfile: Dockerfile.ollama
    container_name: survivalrag-ollama
    ports:
      - "11434:11434"
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:11434/api/tags"]
      interval: 10s
      timeout: 5s
      retries: 30
      start_period: 10s
    restart: unless-stopped

  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: survivalrag-app
    ports:
      - "8080:8080"
    environment:
      - OLLAMA_HOST=http://ollama:11434
      - SURVIVALRAG_MODEL=${SURVIVALRAG_MODEL:-llama3.1:8b}
      - SURVIVALRAG_MAX_CHUNKS=${SURVIVALRAG_MAX_CHUNKS:-5}
      - SURVIVALRAG_RELEVANCE_THRESHOLD=${SURVIVALRAG_RELEVANCE_THRESHOLD:-0.25}
    depends_on:
      ollama:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:8080/api/health"]
      interval: 15s
      timeout: 5s
      retries: 5
      start_period: 30s
    restart: unless-stopped
```

**Design decisions per CONTEXT.md locked decisions:**

1. **Two services** (app + ollama) per DEPL-02. ChromaDB is embedded in the app process, NOT a third container.

2. **Health check configuration:**
   - Ollama: 10s interval, 30 retries, 10s start_period = up to ~5min for model loading on slow hardware. The `/api/tags` endpoint returns the model list only when Ollama is fully ready.
   - App: 15s interval, 5 retries, 30s start_period = reasonable for Python app startup. The `/api/health` endpoint in web.py checks Ollama connectivity and ChromaDB availability.

3. **depends_on with condition: service_healthy** ensures the app container does NOT start until Ollama's health check passes. This is a Docker Compose v2 feature (not available in v1). Combined with the entrypoint's curl polling loop, this provides double safety for startup ordering.

4. **Environment variable passthrough:**
   - `OLLAMA_HOST=http://ollama:11434` -- uses Docker Compose DNS name for inter-container communication. Users can override via .env file to point to an external Ollama instance.
   - `SURVIVALRAG_MODEL=${SURVIVALRAG_MODEL:-llama3.1:8b}` -- reads from host .env file, defaults to llama3.1:8b
   - `SURVIVALRAG_MAX_CHUNKS` and `SURVIVALRAG_RELEVANCE_THRESHOLD` -- passed through with defaults

5. **Port mapping:**
   - `8080:8080` for web UI access per CONTEXT.md locked decision
   - `11434:11434` for direct Ollama API access (useful for debugging, CLI access from host)

6. **restart: unless-stopped** on both services for resilience (restart on crash, don't restart if manually stopped)

7. **No named volumes** per CONTEXT.md locked decision -- knowledge base is immutable, data lives in the container layer.

8. **No explicit networks:** Docker Compose creates a default network automatically. Services reach each other by service name (`ollama`).

9. **container_name** set explicitly for clean `docker ps` output and easy `docker exec` targeting.

**Create docker-compose.gpu.yml** (NVIDIA GPU override file):

```yaml
# SurvivalRAG - NVIDIA GPU Override
# Usage: docker compose -f docker-compose.yml -f docker-compose.gpu.yml up
#
# Requirements:
#   - NVIDIA GPU on Linux host
#   - NVIDIA Container Toolkit installed (nvidia-ctk)
#   - Docker configured with nvidia runtime
#
# Note: Docker on macOS cannot access Apple Metal GPUs.
# For GPU acceleration on Mac, run Ollama natively and set:
#   OLLAMA_HOST=http://host.docker.internal:11434

services:
  ollama:
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
```

**Design decisions per CONTEXT.md locked decisions:**
- Separate override file per CONTEXT.md: "GPU support via separate docker-compose.gpu.yml override"
- Uses `count: all` to pass through all available NVIDIA GPUs
- Uses standard Docker Compose GPU reservation syntax (from Docker Compose v2 spec)
- Only overrides the `ollama` service (the app container doesn't need GPU access)
- Comments document the requirement for NVIDIA Container Toolkit and the macOS limitation (Pitfall 6 from RESEARCH.md)
- Usage command in header comment: `docker compose -f docker-compose.yml -f docker-compose.gpu.yml up`
